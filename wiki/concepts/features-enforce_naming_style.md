## Description
Protobuf identifier naming style enforcement aims to guarantee that protos are round-trippable across languages by default. The Protobuf Editions 2024 design introduces a `features.enforce_naming_style` feature whose `STYLE2024` level enforces strict adherence to the style guide, while the `STYLE_LEGACY` level retains the pre-Edition 2024 enforcement behavior as an explicit opt-out.

The underlying motivation is that the legacy identifier rule — `^[A-Za-z_][A-Za-z0-9_]*$` (the "ASCII identifier" rule) — is too permissive. It allows identifiers with leading underscores, repeated underscores, and mixed casing, which complicates backend transformations between PascalCase, camelCase, snake_case, and SHOUTY_CASE, and can collide with generated private names in target languages.

To resolve this, the stricter schema specifies concrete regex patterns: `([A-Z][a-zA-Z0-9]*)+` for Message/Enum/Service/Method (PascalCase), `[a-z][a-z0-9]*(_[a-z0-9]+)*` for Field names (including extension names) and package components (snake_case), and `[A-Z][A-Z0-9]*(_[A-Z0-9]+)*` for Enum values (SHOUTY_CASE). These patterns are explicitly designed to reject extraneous underscores and to make the casing of ASCII letters consistent. ASCII-only support is a deliberate design trade-off chosen for maximal portability to all target languages. Feature gating controls when this enforcement applies, and the feature is integrated with the broader Protobuf name resolution mechanism.
