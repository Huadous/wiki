## Description
零拷贝是 brpc 中 RdmaEndpoint 数据传输逻辑的核心特性，其核心思想是在数据传输的整个生命周期内避免不必要的数据拷贝。在发送端，所有待发送的数据默认存放在 IOBuf 的 Block 中，这些 Block 仅在收到对端确认接收后才会被释放，其引用则存放于 `RdmaEndpoint::_sbuf` 中，从而确保发送过程中无需将数据从用户缓冲区复制到独立的 RDMA 发送缓冲区。在接收端，零拷贝通过将预提交的接收缓冲区直接置于 IOBuf Block 之中来实现，这些预提交 Block 存放于 `RdmaEndpoint::_rbuf` 内，接收完成后上层可直接消费 Block 中的数据而无需额外的内存拷贝操作。需要注意，接收端预提交的每一段 Block 具有固定大小 `recv_block_size`，因此发送端单个请求的最大尺寸不得超过该值，否则接收端将无法成功接收。

从数据结构层面看，零拷贝的支撑机制是 butil::IOBuf 的非连续零拷贝缓冲设计。IOBuf 通过引用计数管理底层 Block 的生命周期，拷贝 IOBuf 时仅复制管理结构而非数据本身，因此对拷贝的修改不会影响原 IOBuf；将一个 IOBuf append 到另一个 IOBuf 时同样无需复制底层数据，只需转移管理结构的引用。这一"no data copy"特性在源码注释中也得到了明确体现，例如 `buf.append(another_buf);` 操作仅转移管理结构的引用而不复制载荷数据。这种设计使得 IOBuf 在切割、合并等操作上保持高效。性能基准测试显示，在文件读入→切割→合并→写出的工作流中，IOBuf 可达到 240MB/s 至 1519MB/s 的吞吐量，验证了零拷贝设计的实用价值。该机制与 [[concepts/内存注册|内存注册]]、[[concepts/内存池|内存池]] 以及 [[concepts/滑动窗口流控|滑动窗口流控]] 紧密配合，共同支撑 brpc RDMA 高性能数据传输链路。
