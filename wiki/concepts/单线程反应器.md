## Description
单线程反应器是一种典型的事件驱动架构模式，其核心设计为单一线程中的事件循环，统一处理所有 IO 事件的注册、监听和分派。该模式依赖于高效的 IO 多路复用机制（如 epoll、kqueue）来实现非阻塞 IO，通过事件分派机制使单个线程能够同时监控多个文件描述符的 IO 事件。非阻塞 IO 消除了为每个连接分配线程的需求，显著减少了上下文切换和内存开销，从而支持成千上万个并发连接而无需多线程锁竞争的额外负担。Redis 服务器是单线程反应器的经典实现——brpc 的性能测试显示，当使用 50 个 bthread 通过连接池方式同步压测同机 Redis 服务器时，QPS 相比单连接方式大幅回落，同时 Redis 服务器的 CPU 被打满，这验证了单线程反应器的核心瓶颈在于单核 CPU 的计算能力。该架构也解释了连接池模式的性能问题——因为单线程反应器每次只能从一个连接读取一个请求，连接池中的多个连接无法并行利用 CPU 资源，因此单连接共享模式反而性能更优。brpc 通过单连接批量合并请求可大幅提升吞吐量，使 Redis 服务器的 CPU 利用最大化。Nginx 早期版本（master-worker 架构中 worker 为单线程）和 Node.js 也采用类似的单线程事件循环模型，适用于 IO 密集型、计算密集度低的场景，但受限于单核 CPU 的计算能力。brpc 在 HTTP 服务设计中明确不依赖单线程反应器模式，即使处理复杂的 HTTP 消息也不会阻塞响应其他客户端，这是其他基于单线程反应器的 RPC 实现和 HTTP 服务器难以做到的。
