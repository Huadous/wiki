## Description
连接池的核心机制是预先建立并维护一组网络连接，应用程序从池中获取连接执行请求后归还，从而避免每次请求都经历 TCP 三次握手和四次挥手。brpc 的连接池实现支持多路复用和高并发写入，用户可灵活配置连接池的大小和生命周期以平衡资源占用和性能。然而在单线程反应器模型中，由于事件循环每次只能处理一个连接上的一个请求，连接池中的多条连接不仅无法并行处理，反而因频繁的上下文切换和系统调用开销导致性能急剧下降。如 [[concepts/单线程反应器|单线程反应器]] 所述，hiredis 连接池模式在 Redis 上表现不佳的根本原因即在于此。brpc 提供了多种连接类型供用户选择，包括 short（短连接）、pooled（连接池）和 single（单连接），连接的建立和复用对用户透明。在熔断机制方面，当brpc使用连接池模式时，熔断某个节点会导致整个连接池中的所有连接被标记为不可用，且由于brpc的TCP连接被channel共享，所有使用该连接池的channel均无法再使用这些故障连接。用户可通过设置 `ConnectionGroup` 来隔离不同channel的连接池，避免熔断相互影响。
