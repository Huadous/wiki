# 持续接收

## 定义
持续接收（progressive receiving）是指 brpc server 在 HTTP/H2 服务中，针对**超长或无限长请求体（body）**所具备的流式接收能力的描述。brpc 当前版本**不支持**在收齐 HTTP 请求的 header 部分后就立即调用服务回调，因此 brpc server 不适合接收超长或无限长的 body，需要等待完整的请求体到达后才能开始处理。

## 关键特征
- **能力缺失**：brpc server 当前版本不支持在收齐 header 后即触发服务回调，必须等到完整 body 接收完成后才进入服务处理流程。
- **与持续发送不对称**：brpc 在响应方向上具备持续发送（ProgressiveAttachment）能力，可以处理超大或无限长的响应体；但在请求接收方向上不具备对等能力。
- **仅适用于有限 body 场景**：适合处理 body 长度可控、大小适中的 HTTP/H2 请求；不适合需要流式边收边处理的场景。
- **方向性约束**：该限制是 brpc server 端在 HTTP/H2 协议下对**接收方向**的能力边界，发送方向不受此约束。

## 应用
- **不适合的场景**：
  - 客户端需要上传超大文件（如 GB 级别视频、日志批量导入）。
  - 客户端需要持续推送数据流（如实时事件上报、物联网设备遥测数据）。
  - 任何依赖边接收边处理（progressive processing）以降低内存占用的服务端实现。
- **替代方案**：对于需要流式接收的场景，应考虑使用 [[sources/streaming_rpc|Streaming RPC]] 或基于其他流式传输协议（如 gRPC streaming）的方案，而非 brpc 的 HTTP/H2 服务接口。
- **选型依据**：开发者在为 brpc server 设计 HTTP/H2 接口时，需根据请求体大小上限评估是否需要限制请求体长度，并提前在网关或接入层做好拦截。
