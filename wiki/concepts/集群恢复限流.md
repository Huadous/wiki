# 集群恢复限流

## 定义
集群恢复限流是 brpc 在集群宕机恢复阶段提供的客户端限流机制。当集群中所有 server 都处于不可用状态时，集群进入恢复状态。该机制通过控制请求被接受的概率，防止部分 server 在恢复过程中因瞬时流量过载而反复触发熔断。

## 关键特征
- **触发条件**：集群中没有可用 server 时进入恢复状态。
- **概率接受策略**：设服务所有请求所需的最小 server 数为 `min_working_instances`，当前可用 server 数为 `q`，则以 `q / min_working_instances` 的概率接受请求，其余请求被丢弃。
- **退出条件**：若在 `hold_seconds` 时间内 `q` 保持不变，则退出恢复状态，恢复全流量。
- **作用域限制**：目前仅对 `rr` 和 `random` 两种负载均衡算法有效。
- **启用方式**：在 `load_balancer_name` 后追加 `min_working_instances` 和 `hold_seconds` 参数，例如 `load_balancer_name = "rr_min_working_instances=5_hold_seconds=10"`。
- **能力假设**：要求下游各 server 的处理能力相近。
- **不可重试**：被限流丢弃的请求不会被框架重试，调用方可通过 `controller.ErrorCode() == brpc::ERJECT` 判断请求是否被该机制拒绝。

## 应用
- **集群恢复场景**：在集群因故障全部宕机后逐步恢复期间，避免单个刚上线的 server 承接全部流量导致再次过载熔断。
- **过载保护**：作为一种概率性过载保护手段，平滑恢复期的流量分配。
- **配合健康检查**：与 brpc 健康检查机制协同工作，确保恢复过程中流量逐步放量。

- brpc提供了在集群宕机后恢复时的限流机制：当集群中没有可用server时，集群进入恢复状态，假设正好能服务所有请求的server数量为min_working_instances，当前集群可用的server数量为q，则在恢复状态时，client接受请求的概率为q/min_working_instances，否则丢弃；若一段时间hold_seconds内q保持不变，则把流量重新发送全部可用的server上，并离开恢复状态。 — [[sources/client]]
