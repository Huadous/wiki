Editions: Life of a FeatureSet - Summary

来源

  • Original file: editions-editions-life-of-a-featureset.md
  • Ingested: 2026-06-13

核心内容

本文档由 mkruskal-google 撰写并于 2023-08-17 批准，深入讨论了 Protocol Buffers Editions 系统中 FeatureSet 在 protoc、生成器与运行时之间的生命周期管理问题。本文承接 Protobuf Editions Design: Features、Exposing Editions Feature Sets 与 Editions: Runtime Feature Set Defaults 等先期设计，指出其根本缺陷：protoc 无法作为特性解析的通用真相来源，因为它只能通过 import 发现那些被显式导入的生成器特性。

文档推荐采用”每阶段独立解析”方案——在 protoc、各 CodeGenerator 及各运行时中分别实现 Feature Resolution，仅在彼此之间传递 Unresolved Features。该方案虽需跨语言复制解析逻辑，但能最小化代码大小与内存开销，并统一处理 Descriptor Pool 与 Dynamic Messages 场景。文档针对 C++ Generators、Non-C++ Generators、Runtimes Without Reflection、descriptor.proto 引导及动态消息等不同情况分别给出策略，并通过扩展 Conformance Testing 框架（引入 DescriptorConformanceRequest/DescriptorConformanceResponse）保证跨语言一致性。

要点

  • protoc 不能作为特性解析的通用真相来源：它只能通过 import 发现生成器特性，对未被导入的生成器特性无能为力
  • 推荐方案让 protoc、各生成器、各运行时各自独立进行 Feature Resolution，仅在彼此之间传递 Unresolved Features，以最小化代码大小与内存开销
  • C++ Generators 可直接复用现有解析工具并通过 CodeGenerator 虚方法注册自己的特性；Non-C++ Generators 与运行时需复制解析逻辑，但可通过嵌入序列化的 Edition Defaults 避免重复计算
  • Runtimes Without Reflection（如 Java lite、ObjC）由生成器直接把所需已解析特性嵌入自定义对象，无需运行时再做解析
  • descriptor.proto 因无法导入任何特性文件而需被特殊对待，可能需要为引导生成额外的辅助信息
  • Dynamic Messages 支持是设计关键约束：方案必须允许在 bypass protoc 的运行时构造 descriptor 时仍能正确处理 editions
  • 为保证跨语言一致性，需扩展 Conformance Testing 框架，新增 DescriptorConformanceRequest/DescriptorConformanceResponse 来测试任意对 descriptor 的转换
  • 已解析的 FeatureSet 永不对外暴露，仅以包装 API 形式提供，从而借助 Hyrum’s Law 保护内部决策的可演进性
  • 文档还评估了 Default Placeholders、Bidirectional Plugins、Central Feature Registry 等替代方案，但均因代码体积、descriptor pool 支持或版本不一致等问题未被采纳
