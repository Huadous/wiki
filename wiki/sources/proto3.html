要点

  • proto3 是 Protocol Buffers 语言的第三个主要版本,发布于 2016 年。
  • proto3 引入了隐式字段存在性（implicit field presence）概念,在从 proto3 迁移到 editions 时,原有的 implicit 字段会使用 field_presence feature 并设置为 IMPLICIT 值。
  • proto3 文件首行必须通过 syntax = "proto3" 显式声明,否则 Protocol Buffers 编译器默认假定使用 proto2。
  • 每个消息字段必须分配 1 到 536,870,911 之间的唯一 字段编号,其中 19000-19999 为 Protocol Buffers 实现保留,字段编号一旦投入使用便不可更改。
  • 字段编号 1-15 仅占用 1 字节线格式空间,应分配给最常设置的字段以节省空间。
  • proto3 中 singular 字段分 optional（推荐）与 implicit 两种模式,repeated 数值字段默认采用 packed 编码,map 字段用于键/值对类型,消息类型字段自动具有字段存在性。
  • proto3 默认采用无存在性（no presence）模式以简化 API,但通过 optional 标签可以为基本类型字段（数值、字符串、字节、枚举）启用显式存在性（explicit presence）跟踪。
  • 从 v3.15.0 版本开始,proto3 中 optional 字段的显式存在性跟踪功能默认启用;在 v3.12.0 之前,需要使用 --experimental_allow_proto3_optional 编译标志才能启用该功能。
  • 根据 proto3 语法规则,所有枚举类型都必须包含一个映射到 0 的枚举值,通常命名为 UNKNOWN。
  • 删除字段 时必须将对应字段编号加入 保留列表（也建议保留字段名称）,否则复用编号会导致解析错误、PII 泄漏或数据损坏。
  • protoc 编译器为 C++、Java、Kotlin、Python、Go、Ruby、Objective-C、C#、PHP 等目标语言分别生成不同格式的数据访问代码。
  • singular 字段在线格式字节中出现多次时,解析器仅保留最后一次出现的实例,体现 Last One Wins 语义。
  • 推荐使用 optional 字段标签以获得与 Protobuf Editions 和 proto2 的最大兼容性。
  • 相比 proto2,proto3 简化了语法并修改了字段存在性的默认行为,消除了对 required 字段的支持以及默认值设置的复杂性。
  • 相较于 proto2 默认追踪字段存在性,proto3 默认不追踪 字段存在性。
  • proto3 通过 optional 关键字或 wrapper 类型两种方式支持 字段存在性;被标记为 optional 的基本类型 singular 字段（数值、字符串、字节、枚举）会像 proto2 一样追踪 字段存在性,而 implicit singular 字段则继续省略存在性信息。
  • proto3 描述符已使用 [[concepts/label-optional|LABEL_OPTIONAL]] 描述不追踪存在性的 singular 字段,因此不能直接复用 proto2 的 [[concepts/label-optional|LABEL_OPTIONAL]] 描述符来表示存在性——大量现有反射代码已假设 proto3 中的 [[concepts/label-optional|LABEL_OPTIONAL]] 不包含存在性信息。
  • 在 features.enum_type 这一 feature 方面,proto3 默认采用 OPEN（open enums）枚举模式。
  • 在 features.field_presence 这一 feature 方面,proto3 默认为 IMPLICIT字段存在性;除非字段带有 optional 标签,此时其行为类似于 EXPLICIT。
  • proto3 在符号可见性方面默认采用 EXPORT_ALL。
  • proto3 在命名风格方面默认采用 STYLE_LEGACY。
  • 需要注意的是,features.enum_type 这一 feature 对 proto3 文件不产生影响。
  • proto3 的语言指南是一个独立的文档,但 editions 指南对其向 Protobuf Editions 迁移时的行为做了相应说明。
  • 在 Edition Zero 推出之前,proto3 与 proto2 并存使用。
  • Edition Zero 通过 features 机制把 proto2 与 proto3 的差异统一到一组良定义的默认值上,因此 proto3 在 Edition Zero 的统一工作中扮演重要角色。
  • 在过渡期内,protoc 必须能够同时解析 proto3、proto2 和 editions 文件。
  • Protobuf 团队将提供一个工具,能够以完全兼容的方式将 proto2 和 proto3 文件升级到 Edition Zero,从而帮助 schema producers 完成迁移。
  • 对于新发布的 .proto 文件,schema producers 应直接使用 Edition Zero 的默认值,而非继续沿用 proto3 语法。
  • proto3 是 Protobuf 的第三代语法标准,在 Protobuf Editions 出现之前是 Protobuf 的最新语法版本。
  • Editions 机制旨在统一替换 proto2 和 proto3,提供更灵活、可演进的 features 系统。
  • 与 proto2 类似,从 proto3 到 Editions 的迁移也是通过 Editions upgrader 工具（即 protoc --upgrade-edition）完成的。
  • 该工具会为文件添加适当的 edition 声明（如 edition = "2023";）和 option features.* = ...; 选项,使其保留原始行为。
  • Edition Zero 的迁移涉及同时处理 proto2 和 proto3 这两种旧语法。
  • 从 proto2 和 proto3 迁移到 Editions 属于大规模变更（Large-scale Change）,且这一变更可以完全自动化,无需破坏性改动。
  • 在 Editions 设计早期阶段,曾讨论过将 proto2 和 proto3 作为”特殊 edition”的方案,但该方案的具体形态和必要性当时并未确定。
  • 提案建议将 proto3 视为 Legacy Syntax Editions 中的特殊 edition,具体取值为 EDITION_PROTO3 = 999。
  • 在 proto3 中,optional 关键字会将对应的 field_presence feature 设置为 EXPLICIT,即设置显式存在性。
  • proto3 的语法元素（如 optional 关键字设置的 EXPLICIT presence）与 Editions 的 feature inference 之间存在映射关系,需要通过专门的 feature inference 逻辑来正确推断出对应的 feature 行为。
  • proto3 引入了对 proto2 的多项更改,包括字段默认值的设定（如 int32 的 0、string 的空字符串等）以及移除 required 字段。
  • Edition Zero 保留了所有 proto2/proto3 行为,这使得 proto3 成为初始版本旨在向后兼容的两个前身标准之一。
  • proto3 与 proto2 一起代表了 Editions 系统必须保持兼容的现有行为。
  • Editions 系统的设计目标包括与 proto3 现有行为的兼容性,使得现有用户向新系统的迁移是无缝的。
  • 在 UTF8 验证等 feature 建模方式的讨论中,由于 Edition Zero 保留了所有 proto2/proto3 行为,因此无论选择何种 feature 表达方式,都不会产生功能性差异,讨论的焦点仅在于应使用哪些 feature 来控制这些行为。
  • proto3 是 Protobuf 的第三版语法,是当前推荐使用的版本。
  • 在 JSON 处理方面,proto3 默认完全验证 JSON 映射的唯一性:protoc 会在解析时检测到任何 JSON 冲突并直接报错。
  • 可通过 deprecated_legacy_json_field_conflicts 选项禁用 proto3 的这一严格检查,使其退化为尽力而为模式（即 LEGACY_BEST_EFFORT 行为）。
  • 本次设计提案建议将 proto3 默认迁移到 ALLOW 状态,以调整其 JSON 字段名冲突的处理策略。
