brpc内存管理：ResourcePool与ObjectPool - Summary

来源

  • Original file: memory_management.md
  • Ingested: 2026-06-13

核心内容

本文深入讨论了 brpc 框架中的内存管理机制，重点介绍 ResourcePool<T> 和 ObjectPool<T> 两个核心组件。多线程内存分配需要在减少线程间竞争与降低空间浪费之间权衡，通用方案如 tcmalloc 和 jemalloc 虽成熟但对底层性能敏感的应用仍不够用。brpc 利用大多数结构等长的特点，通过批量分配与归还、thread-local free block 和全局 free block 的层级化设计，实现了高效的 等长对象分配。ResourcePool 返回偏移量以支持作为 64 位 id 的一部分（用于生成 bthread_t），ObjectPool 则直接返回对象指针。bthread 的栈管理通过 mmap 分配并配合 guard page 检测溢出，提供 NORMAL、SMALL、LARGE 三种规格。文章也对比了 goroutine 的 segmented stacks 和 变长连续栈 方案，阐述了 bthread 选择等长栈的原因。

关键实体

  • brpc — Apache 开源的高性能 RPC 框架，源自百度，本文讨论的内存管理组件均属于该框架
  • tcmalloc — Google 开发的高性能多线程内存分配器，文中作为通用方案参照
  • jemalloc — Facebook 主导开发的高性能通用内存分配器，文中作为通用方案参照

关键概念

  • ResourcePool<T> — brpc 的核心内存分配模板类，返回偏移量而非指针，专为等长对象设计
  • ObjectPool<T> — ResourcePool 的变种，直接返回对象指针
  • 等长对象分配 — brpc 内存管理的核心优化假设，大幅简化分配过程
  • 内存池 — 通过批量分配与层级缓存降低竞争与开销的技术
  • bthread — brpc 的用户态线程（协程），平均创建耗时小于 200ns
  • bthread_t — 由 32 位版本号 + 32 位偏移量组成的 bthread 标识符
  • taskmeta — bthread 的底层数据结构，所有 TaskMeta 由 ResourcePool 分配
  • ABA问题 — 无锁编程中的经典现象，通过 bthread_t 中的版本号机制缓解
  • socketid — 采用与 bthread_t 相同的生成方式（版本号+偏移量）
  • bthread_id_t — 采用与 bthread_t 相同的生成方式
  • bthread 栈管理 — 按 NORMAL/SMALL/LARGE 三档管理 bthread 栈
  • guard page — 用于检测栈溢出的保护性内存页（4K）
  • BTHREAD_ATTR_NORMAL — 默认 1M 栈大小，server 端默认使用
  • BTHREAD_ATTR_SMALL — 默认 32K 栈大小，适合大量小任务
  • BTHREAD_ATTR_LARGE — 与 pthread 同栈大小，不做 caching
  • segmented stacks — goroutine 1.3 之前的栈管理方案，存在 hot split 问题
  • 变长连续栈 — goroutine 1.3 之后的栈管理方案，类 vector resizing
  • hot split — 分段栈中因栈边界频繁扩张/收缩导致的性能问题

要点

  • 多线程内存分配的核心权衡在于线程间竞争少与浪费空间少之间，通用 malloc 方案对底层性能敏感的应用仍不够用
  • brpc 利用”大多数结构等长”的属性，通过 ResourcePool 和 ObjectPool 做极致优化，但明确不鼓励用户在程序中滥用
  • ResourcePool 通过 thread-local free block → 全局 free block → 批量申请的三级机制实现低竞争分配，并返回偏移量（可作为 64 位 id 的一部分）
  • bthread_t 由 32 位版本号 + 32 位偏移量组成，通过版本号解决 ABA问题；该 id 生成方式在 brpc 中广泛应用（包括 SocketId、bthread_id_t）
  • bthread 创建平均耗时小于 200ns，底层结构 taskmeta 由 ResourcePool 分配
  • bthread 栈分为 NORMAL（1M）、SMALL（32K）、LARGE 三档，使用 mmap 分配并设置 guard page 检测溢出
  • 对比 goroutine 的 segmented stacks 和 变长连续栈 方案，bthread 因主要运行在 64 位平台、虚存空间庞大，决定暂不引入变长栈
